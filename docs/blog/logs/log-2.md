# On Building Small Things

There's a specific satisfaction that comes from finishing something small and complete.

Not a feature inside a larger system. Not a ticket in a backlog. A whole thing — with edges and a purpose and a moment where it's just done.

I've been thinking about why that feeling is rare in most engineering jobs. The scope is always bigger than a sprint. The product is never finished, only iterated. And "done" gets redefined every quarter.

So lately I've been deliberately building small things on the side:

- A script that formats my notes the way I actually want them
- A tiny CLI tool for something I was doing manually every week
- This website

None of them are impressive. All of them taught me something.

## What small things force you to do

When scope is limited, you can't defer decisions. You have to pick a data structure, a file format, an interface — and ship with it. There's no "we'll refactor this later." There is no later.

That constraint is clarifying. It forces taste. You end up caring about the thing being right more than the thing being scalable.

## The value isn't the artifact

The script I wrote for my notes is probably not useful to anyone else. That's fine. The value was in building the mental model of what I actually wanted — and then making it real.

Most side projects fail because people try to build something useful to others before they've built something true to themselves.

Start smaller. Finish it. Then do it again.
